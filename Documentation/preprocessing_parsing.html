Icarus Verilog Preprocessing and Parsing Pipeline

A concise technical overview of how ivlpp and the compiler (ivl) work together

Overview

Icarus Verilog compiles Verilog/SystemVerilog sources by first preprocessing them with ivlpp to produce a single flattened stream, then lexing and parsing that stream into a rich parse form (pform), elaborating it into a netlist, optimizing, and finally emitting target code (commonly VVP bytecode) for simulation or downstream tools.

Preprocessing (ivlpp)

ivlpp is a standalone preprocessor invoked by iverilog. It processes directives and macros, and manages include files and conditional compilation, optionally emitting `line directives to preserve file/line mapping.

Inputs and Options

  • -f filelist: list of sources; -F flags file: defines/includes.
  • -I include dirs; -K keyword macros (pass-through directives).
  • -L emit `line directives; -W redef warnings.
  • -P/-p load/dump precompiled defines; -o output path.

Include Handling

  • Search order: current file directory (unless disabled via relative mode), then -I paths; absolute paths are used as-is.
  • On include: push a new file buffer onto the include stack; optionally emit `line and write dependency entries when requested.
  • Comments on the include line are preserved and emitted after the included content.

Macros

  • Definition: `define with optional formal arguments and defaults (e.g., name(arg=default)).
  • Body collection: strips comments, trims whitespace, handles line continuation (\).
  • Formal substitution: occurrences of argument names in the body are rewritten to an internal argument mark + index to splice actuals efficiently at expansion time.
  • Expansion: argument parsing handles nested calls with balanced parentheses; the expansion is re-lexed by pushing a string “input” on the include stack.
  • Stringification (Icarus extension): ``NAME turns expansion into a string, escaping quotes and backslashes appropriately.
  • Magic macros: __LINE__ and __FILE__ computed from the outermost real file context.
  • Keyword macros: certain directive-like tokens are defined to pass through untouched (e.g., `timescale).

Conditional Compilation

  • Supports `ifdef, `ifndef, `elsif, `else, `endif with a stack to ensure proper matching.
  • True branches pass through; false branches are suppressed while maintaining line accounting; mismatches produce diagnostics.

Strings and Comments

  • Strings are emitted verbatim; macro expansion does not occur inside normal strings.
  • C-style and pragma comments are passed through; directives nested in comments are ignored. Pragmas allow macro expansion but forbid directive keywords.

Directives

  • `line: updates current file/line and can be echoed to output (-L).
  • `pragma: syntax-checked (must have a name) and passed through.

Precompiled Defines

  • -p writes defines in a compact binary-safe format; -P loads them.
  • Redefinition warnings controlled via -W flags (redef-all, redef-chg).

VHDL Pass-through

  • If configured, .vhd/.vhdl files are preprocessed via vhdlpp and their output is treated as source.

Parsing and AST (pform)

After preprocessing, the compiler (ivl) uses a flex scanner (lexor.lex) and a bison grammar (parse.y) to tokenize and parse the source into a rich parse form (pform). This is an AST-like structure comprising typed C++ objects with file/line annotations.

Parsed Representation

  • Expressions: PExpr hierarchy (e.g., PENumber, PEUnary, PEBinary, PEConcat, PECallFunction, PEIdent, PETernary, PECast*).
  • Statements/Processes: Statement hierarchy (PAssign, PAssignNB, PDelayStatement, PEventStatement, PCase, PCondit, PBlock, loops, tasks, calls).
  • Design structure: Module, ports (Module::port_t), nets/wires (PWire), UDPs, packages, classes, typedefs, generates, specify blocks (parsed; limited support).
  • Names: pform_name_t and PEIdent capture hierarchical identifiers, part/bit selects, and package-resolved names.

Scope and Attributes

  • Scopes are pushed/popped for modules, functions, tasks, blocks, generates, packages, classes, etc.
  • ANSI/non-ANSI ports are recorded with direction, kind, data type, and binding expressions or concatenations.
  • Attributes ((* ... *)) are collected and bound to nodes.
  • All nodes get file/line info via FILE_NAME macros for diagnostics.

Post-Parse Phases

  • Elaboration: scope and parameter resolution builds a NetScope tree; the structural and behavioural netlist is generated (see elab_*.cc, netlist.*).
  • Optimization: transformations like constant propagation, dead-code removal, combinational reduction (cprop.cc, expr_synth.cc, nodangle.cc, etc.).
  • Code Generation: emit() walks the netlist and calls a target backend (tgt-vvp, tgt-verilog, tgt-vhdl, tgt-blif, etc.).

Introspection

  • -P <path>: dump the pform (AST-like) for debugging.
  • -N <path>: dump the elaborated/optimized netlist pre-codegen.
  • -g*: feature flags (e.g., -gspecify, -g2005-sv, -gicarus-misc).

Putting It Together

  1. Preprocess with ivlpp: expand macros/conditionals, resolve includes, preserve line mapping.
  2. Lex & Parse with ivl: build pform nodes with typed C++ objects and file/line metadata.
  3. Elaborate: resolve scopes and parameters, generate netlist.
  4. Optimize: apply generic, target-independent passes.
  5. Emit target code: VVP (default) or other backends for simulation or downstream consumption.
Note: Specify blocks are parsed but disabled by default; enable with -gspecify for limited support. SystemVerilog support is partial and controlled by generation flags.
